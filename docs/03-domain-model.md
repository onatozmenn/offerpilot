# Domain Model

## Experiment

An experiment defines one selection problem.

Required fields:

- UUID identifier.
- Unique slug and display name.
- Status: `draft`, `running`, `paused`, or `completed`.
- Policy kind: `random` or `segmented_epsilon_greedy`.
- Exploration epsilon in the inclusive range `[0, 1]` when applicable.
- Current policy version, starting at one.
- Created and updated UTC timestamps.

Only `running` experiments accept decisions and feedback. Resetting an experiment creates a new experiment or run; historical decisions are never silently deleted.

Policy kind, epsilon, reward mapping, and approved context schema are immutable after experiment creation. Changing one creates a new experiment so historical decisions retain their original semantics. A demo experiment is created in `running` state even when no simulation run is active.

## Offer

An offer is an action available to a policy.

Required fields:

- UUID identifier.
- Experiment identifier.
- Stable slug.
- Fictional merchant name.
- Title and short description.
- Category: `travel`, `dining`, `wellness`, `home`, `technology`, or `entertainment`.
- Active flag.

An experiment needs at least two active offers. Offer slugs are unique within an experiment. An inactive offer cannot be selected but remains attached to historical decisions.

## Session Context

The context is an immutable value object containing only documented enum values:

- Device class.
- Daypart.
- Category affinity using the same closed category set as offers.
- Visitor type.

The canonical segment key joins normalized values in a fixed documented order. Unknown or missing values are rejected instead of silently assigned to a default segment.

## Decision

A decision is the immutable record of one policy choice.

Required fields:

- UUID identifier generated by the server.
- Experiment and selected-offer identifiers.
- Context and canonical segment key.
- Eligible offer identifiers in deterministic order.
- Probability distribution for every eligible offer.
- Selected propensity greater than zero and at most one.
- Policy kind and policy version.
- Policy selection latency in non-negative microseconds.
- Optional simulation-run identifier.
- UTC creation timestamp.

The distribution must contain exactly the eligible offers and sum to one within `1e-9`. The selected propensity must equal the selected offer's recorded probability.

## Outcome

An outcome is terminal feedback for one decision.

Required fields:

- Client-generated UUID event identifier for idempotency.
- Decision identifier.
- Outcome kind: `ignored`, `clicked`, or `converted`.
- Server-derived reward.
- UTC occurrence and receipt timestamps.
- Consecutive policy-application version reserved and applied by the server.

One decision accepts at most one terminal outcome. Repeating the same event is idempotent. A different event for a decision that already has an outcome returns a conflict. Feedback may arrive after newer decisions; selection version is retained for audit while application version follows feedback acceptance order.

## Policy State

Policy state is a versioned snapshot scoped to one experiment.

The random policy has no learned parameters but advances its application version for each accepted outcome. Segmented epsilon-greedy stores, for every segment and offer, selection count, cumulative reward, and empirical mean. New segment-offer pairs start with documented optimistic priors to avoid permanent cold-start starvation.

## Simulation Run

A simulation run records a reproducible traffic generation session:

- UUID identifier.
- Experiment identifier.
- Seed.
- Requested rate and maximum events.
- Status: `starting`, `running`, `stopping`, `completed`, `failed`, or `cancelled`.
- Started, stopped, and updated UTC timestamps.
- Generated decision, outcome, and error counts.
- Observed reward sum plus random-policy and oracle expected-reward sums, all simulation-only.

Only one in-process simulation may run for an experiment. Stop is idempotent.

## Reward Semantics

Reward is bounded to `[0, 1]` and is derived on the server:

| Outcome | Reward |
| --- | ---: |
| `ignored` | 0.00 |
| `clicked` | 0.25 |
| `converted` | 1.00 |

This is a demo utility function, not a claim about real commercial value. Changes require a new experiment version because historical rewards must retain their original meaning.

## Global Invariants

- Identifiers are UUIDs and timestamps are UTC.
- Policy randomness is generated from an injected source.
- No domain object contains PII or protected attributes.
- Historical decisions and outcomes are append-only.
- Floating-point values must be finite; `NaN` and infinity are always invalid.
- A policy update is attributable to one accepted outcome and one resulting version.
- Application versions are consecutive by accepted-feedback order; a decision's selection version may be older without invalidating its outcome.
